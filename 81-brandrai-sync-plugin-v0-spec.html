BrandrAI Sync Plugin v0 — WordPress + Bricks Builder

Technical design for the WP plugin that connects a customer's WordPress site to their BrandrAI account, applies brand system + generated pages via REST API, and keeps them synced. This v0 targets Bricks Builder specifically — the cleanest data model of the major WP page builders.

1 · Goals + non-goals

Scope lines for v0.

v0 MUST do
  • Install via WP admin or .zip upload
  • OAuth to a BrandrAI account
  • Detect Bricks Builder is active; warn if not
  • Pull brand system from BrandrAI, apply to bricks_theme_styles + bricks_color_palette options
  • Trigger CSS regeneration via wp bricks regenerate_assets
  • Allow user to select pages to generate (via BrandrAI dashboard OR WP admin)
  • Receive page JSON from BrandrAI, write to _bricks_page_content_2 post meta
  • Log sync events to admin screen for debugging
v0 will NOT do
  • Support other builders (Breakdance, Elementor) — v1+
  • Two-way sync (pull WP edits back to BrandrAI) — v2
  • Staging preview before push — v2
  • Versioning + rollback — v2
  • White-label for agencies — v3
  • Multi-site support from a single plugin install — v3
2 · UX flow

What a customer experiences.

  1. Install. Customer uploads brandrai-sync.zip via WP admin → Plugins → Add New → Upload. Clicks Activate.
  2. Onboarding screen. WP admin → BrandrAI tab. Welcome screen with a single “Connect to BrandrAI” button.
  3. OAuth. Button redirects to app.brandrai.com/oauth/authorize?site_url=.... User logs in (or signs up). Consent screen: “BrandrAI Sync wants to apply brand and sync pages to [site.com].” Approve.
  4. Callback. BrandrAI redirects back to site with auth code. Plugin exchanges for access + refresh token. Stores encrypted in wp_options.
  5. Detection. Plugin checks if Bricks is active. If yes, shows “Ready.” If no, shows warning: “BrandrAI Sync currently supports Bricks Builder. Install Bricks first.”
  6. First sync. Plugin fetches brand system from BrandrAI API. Writes bricks_theme_styles + bricks_color_palette. Runs wp bricks regenerate_assets. Customer sees a success toast: “Brand applied. 20 color tokens + typography scale + spacing system installed.”
  7. Page generation. Customer opens BrandrAI dashboard → Sites tab → picks the connected site → selects pages to generate (Homepage, Pricing, Cannabis, etc.). Clicks Generate + Push.
  8. BrandrAI generates. Server-side Python produces Bricks-format JSON for each page. POSTs to /wp-json/brandrai/v1/pages on the connected site.
  9. Plugin receives. Plugin endpoint authenticates the request (HMAC signature), creates or updates WP pages, writes _bricks_page_content_2 meta, returns success.
  10. Customer verifies. Customer refreshes WP admin → Pages. New pages exist. Opens any one in Bricks builder. All elements on-brand, content populated. Makes manual tweaks as needed. Publishes.

Total time from install to first branded page live: ~5-10 minutes. Most of that is OAuth + page selection; the actual sync is seconds.

3 · Architecture

Two systems, one pipe.

┌─────────────────────────────┐              ┌──────────────────────────────┐
│   BrandrAI (SaaS backend)   │              │ Customer WordPress + Bricks  │
│   Laravel / PHP / Python    │              │                              │
├─────────────────────────────┤              ├──────────────────────────────┤
│                             │              │                              │
│   Brand guide storage       │─ OAuth ───►  │  BrandrAI Sync plugin        │
│   Wireframe generator       │  token       │   ├── REST endpoint          │
│   Page template library     │              │   ├── Admin UI               │
│                             │─ POST ─────► │   ├── WP hooks               │
│   /oauth/authorize          │  signed JSON │   └── Auth + token store     │
│   /oauth/token              │  webhook     │                              │
│   /api/brand/{site_id}      │              │  Bricks Builder              │
│   /api/pages/generate       │              │   ├── bricks_theme_styles   │
│                             │  ◄─ GET ─────│   ├── bricks_color_palette  │
│                             │  brand pull  │   └── _bricks_page_content_2│
└─────────────────────────────┘              └──────────────────────────────┘

Two endpoints do the heavy lifting:

  • BrandrAI → Plugin (push): When brand or pages regenerate, BrandrAI signs a JSON payload with HMAC and POSTs to plugin's endpoint. Plugin verifies signature, applies changes.
  • Plugin → BrandrAI (pull, minimal): On initial connect + periodically (daily), plugin fetches current brand system to stay in sync. Mostly a safety net in case webhooks miss.

Why push, not pull: push is faster (no polling), more reliable (event-driven), cheaper (no wasted requests). Pull is just a backup.

4 · Plugin components

PHP file layout.

brandrai-sync/
├── brandrai-sync.php           # Plugin header + bootstrap
├── includes/
│   ├── class-brandrai-auth.php       # OAuth + token storage
│   ├── class-brandrai-rest.php       # REST endpoints
│   ├── class-brandrai-bricks.php     # Bricks-specific writers
│   ├── class-brandrai-sync.php       # Core sync orchestration
│   ├── class-brandrai-admin.php      # WP admin UI
│   └── class-brandrai-logger.php     # Structured event logging
├── assets/
│   ├── admin.css
│   └── admin.js
├── templates/
│   └── admin-*.php             # Admin UI templates
├── vendor/                     # Composer dependencies (if any)
├── composer.json
└── readme.txt                  # WP plugin repo metadata

class-brandrai-auth.php

Handles OAuth flow: redirect to BrandrAI authorize, receive callback, exchange code for tokens, store encrypted, refresh on expiry. Uses WP's sodium or openssl for at-rest encryption.

class-brandrai-rest.php

Registers /wp-json/brandrai/v1/ namespace. Endpoints: POST /brand, POST /pages, POST /pages/{id}, GET /status, POST /disconnect. All protected by HMAC signature verification using shared secret from OAuth.

class-brandrai-bricks.php

Bricks-specific writers. Methods: apply_theme_styles($payload), apply_color_palette($payload), upsert_page($page_id, $elements), create_page($title, $elements), regenerate_css(). All operate via update_option() and update_post_meta(). Triggers wp bricks regenerate_assets via do_action.

class-brandrai-sync.php

Orchestration. On webhook receive, dispatches to appropriate writer. Wraps in transaction-like try/catch. Logs to class-brandrai-logger.php. Returns status back to BrandrAI.

class-brandrai-admin.php

WP admin UI. “BrandrAI” menu item. Four screens: Connect (OAuth), Status (connection + last sync), Logs (recent events), Settings (preferences). Uses WP nonces + capabilities for every form action.

class-brandrai-logger.php

Structured logging. Custom DB table wp_brandrai_sync_log with timestamp, event_type, payload, status, error. Visible in admin Logs screen. Auto-purged after 30 days.

5 · API contract

What BrandrAI sends.

POST /wp-json/brandrai/v1/brand

Apply full brand system. Payload:

{
  "brand_id": "brandrai-self",
  "version": "2026-04-22",
  "theme_style": { ... full Bricks theme style JSON ... },
  "color_palette": { ... full Bricks color palette JSON ... },
  "signature": "sha256=abc123..."
}

POST /wp-json/brandrai/v1/pages

Create or update one or more pages. Payload:

{
  "pages": [
    {
      "slug": "pricing",
      "title": "Pricing",
      "wp_id": null,   // null = create new; id = update existing
      "bricks_elements": [ ...flat element array... ]
    },
    { ...more pages... }
  ],
  "signature": "sha256=..."
}

GET /wp-json/brandrai/v1/status

Health check + connection info. Response:

{
  "connected": true,
  "brand_version": "2026-04-22",
  "last_sync": "2026-04-22T20:14:32Z",
  "bricks_version": "1.12.0",
  "bricks_active": true,
  "wp_version": "6.5",
  "plugin_version": "0.1.0"
}

HMAC signature verification

Every incoming request must include:

X-BrandrAI-Signature: sha256=<hex_digest>
X-BrandrAI-Timestamp: 1714060800

signature = hmac_sha256(shared_secret, timestamp + "." + body)

Plugin verifies signature, rejects if timestamp > 5 min old (replay protection). Shared secret established during OAuth handshake, rotatable via re-auth.

6 · Auth + security

Don't get this part wrong.

  • OAuth 2.0 standard flow. Authorization Code grant. Redirect URI = {site_url}/wp-admin/admin.php?page=brandrai-sync&action=callback.
  • Tokens stored encrypted. Access token + refresh token + shared HMAC secret encrypted with sodium using a key derived from WP_AUTH_KEY.
  • HMAC on every webhook. No exceptions. Unsigned requests return 401.
  • Timestamp replay protection. Reject requests > 5 min old.
  • Capability checks. Only manage_options users can connect, disconnect, or reconfigure. All admin actions use WP nonces.
  • Rate limiting. Max 10 writes per minute per site to prevent accidental runaway syncs. Exceeded = 429.
  • No PII beyond site identity. We never receive or store the customer's visitors' data. Only brand assets + their own page content.
  • Disconnection is clean. Disconnect = delete stored tokens + notify BrandrAI to revoke. Pages + brand styles stay in place (don't surprise-delete customer content).

8 · Bricks integration specifics

The actual code that matters.

Applying theme styles

// class-brandrai-bricks.php
public function apply_theme_styles( $payload ) {
    // $payload is the theme_style array from BrandrAI
    $existing = get_option( 'bricks_theme_styles', array() );
    $brandrai = array( 'brandrai-default' => $payload['theme_style'] );

    // Merge — keep customer's custom styles; override ours by id
    $merged = array_merge( $existing, $brandrai );
    update_option( 'bricks_theme_styles', $merged );

    // Also update color palette
    $existing_palette = get_option( 'bricks_color_palette', array() );
    $merged_palette = array_merge( $existing_palette, array(
        'brandrai' => $payload['color_palette']
    ));
    update_option( 'bricks_color_palette', $merged_palette );

    // Trigger CSS regen
    $this->regenerate_css();
}

private function regenerate_css() {
    // Bricks' own regenerate command
    if ( class_exists( '\Bricks\Assets' ) ) {
        \Bricks\Assets::regenerate_css_file_global_classes();
        \Bricks\Assets::regenerate_css_file_theme_styles();
    }
}

Upserting a page

public function upsert_page( $slug, $title, $elements ) {
    // Find existing page by slug
    $existing = get_page_by_path( $slug );
    if ( $existing ) {
        $post_id = $existing->ID;
    } else {
        // Create new page
        $post_id = wp_insert_post( array(
            'post_title'   => $title,
            'post_name'    => $slug,
            'post_type'    => 'page',
            'post_status'  => 'draft',
            'post_content' => '',
        ));
    }

    // Write Bricks element tree to post meta
    update_post_meta( $post_id, '_bricks_page_content_2', $elements );

    // Mark page as edited with Bricks (not WP editor)
    update_post_meta( $post_id, '_bricks_editor_mode', 'bricks' );

    // Clear cache for this page
    if ( function_exists( 'bricks_clear_post_cache' ) ) {
        bricks_clear_post_cache( $post_id );
    }

    return $post_id;
}

Respecting customer edits

If a customer has edited a page in Bricks since the last sync, we should NOT blindly overwrite. Options:

  • Default: compare last-synced hash vs current hash. If different, refuse the push + return 409. BrandrAI dashboard shows “This page was edited since last sync — overwrite?”
  • Force mode: customer explicitly opts in to overwrite via dashboard checkbox.
  • Diff mode (v2): show element-level diff before committing.

For v0: default behavior is safe overwrite refusal. User must confirm overwrite from dashboard.

9 · Testing + QA

What has to pass before shipping.

  • Unit tests for auth, HMAC verification, Bricks writers, error cases
  • Integration tests against a fresh WP + Bricks install (Docker-based CI)
  • Manual test matrix: Bricks 1.10, 1.11, 1.12 + WP 6.4, 6.5, 6.6
  • OAuth flow on three different hosts (shared, WP Engine, Kinsta)
  • Sync a 20-page site, confirm no memory/timeout issues
  • Disconnect + reconnect flow leaves no orphan data
  • Plugin upgrade flow (v0 to v0.1) preserves customer connections
  • Wordfence + iThemes Security compatibility (the two most common WP security plugins)
  • Multisite (subdomain + subfolder) compatibility check
10 · Distribution

How customers get it.

  • v0: direct .zip download from app.brandrai.com after OAuth signup. Avoids WordPress.org review delays.
  • v0.2: submit to WordPress.org plugin directory. Expect 2-6 week review. Free distribution tier.
  • v0.3: listed on Bricks Builder's “Community Extensions” page (if they accept) for credibility.
  • v1: listed on Codeable + other WordPress-adjacent marketplaces.

Free vs paid: plugin itself is free. Functionality requires a BrandrAI account (which can be free Starter, paid Growth, etc.). Freemium funnel.

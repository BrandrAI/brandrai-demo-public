Cross-Builder Brand Cascading and Page-Content Generation for Third-Party Website Builders — Brand-New Filing

Technical description of a computer-implemented system and method that applies a design-token-based brand system to the native configuration format of one or more third-party website builders (including, without limitation, Breakdance, Bricks Builder, Elementor, Oxygen, Kadence, Gutenberg, Divi, Beaver Builder, and Shopify theme architecture), producing import-compatible page content that references the target builder's native CSS-variable infrastructure such that subsequent changes to the brand system propagate automatically to all generated content.

1. Proposed title

“Method and System for Applying a Design-Token-Based Brand System to Third-Party Website-Builder Configuration Files via Builder-Specific Adapter Architecture with Automatic Propagation”

Counsel: alternative working titles include “Cross-Platform Brand Consistency Through Design-Token-Aware Page Generation,” “Method for Generating Page Content in Third-Party Website-Builder Native Format Using Shared Design Tokens,” or simply “BrandrAI Sites.”

2. Field of the invention

This invention relates to systems and methods for applying a unified visual brand identity across websites built with heterogeneous website-builder software. More specifically, it pertains to the combination of design-token-based brand systems, builder-specific format adapters, token-aware page-content generation, and automatic cascade mechanisms whereby brand changes propagate to generated content without per-element re-styling.

3. Background — state of the art

Multiple partial solutions exist for brand consistency and page generation, but none combines the necessary elements to apply a brand system to arbitrary pre-existing third-party website-builder configurations:

Design-token tools (Figma Tokens, Specify.app, Style Dictionary)Read/export design tokens in structured format (typically DTCG-compatible)No CMS / page-builder integration; stops at code export; no page content generation

The specific gap: no existing system accepts (a) a design-token-based brand guide from a customer and (b) an export or representation of the customer's existing website-builder configuration, and produces (c) page content in that builder's native import-compatible format with (d) element styling bound to the builder's own CSS-variable infrastructure, such that (e) subsequent changes to the brand guide propagate automatically without per-element regeneration, across (f) multiple different builder vendors through a pluggable adapter architecture.

The need is particularly acute for brands operating across multiple sites (multi-state cannabis operators, ecommerce brands with regional variants, agencies managing client portfolios) who cannot achieve brand consistency today without either forcing all sites onto one builder (infeasible for established brands) or manually reconfiguring each builder's design system (time-prohibitive and error-prone).

4. Brief summary of the invention

A computer-implemented method for applying a design-token-based brand system to a third-party website-builder configuration, comprising:

  1. Receiving a brand system comprising design tokens including at least one of: color tokens, typography tokens, spacing tokens, border-radius tokens, layout tokens, and content-voice tokens;
  2. Receiving a representation of a target website-builder's configuration, wherein said representation may take the form of an exported configuration file, a direct database query against the builder's storage, or a REST API response from the builder's programmatic interface;
  3. Selecting, from a library of builder-specific adapters, an adapter corresponding to the identified target builder, wherein each adapter encodes the target builder's native element schema, property naming conventions, nesting semantics, and CSS-variable binding syntax;
  4. Generating, using the selected adapter, page-content configuration data in the target builder's native format, wherein generated elements reference the brand system's design tokens through the builder's native CSS-variable infrastructure rather than hard-coded style values;
  5. Producing an import-compatible output payload in the format accepted by the target builder's native import mechanism (whether by file, REST API, WP-CLI command, database write, or equivalent);
  6. Whereby changes to the brand system propagate to all generated pages automatically through the shared token-variable references without requiring per-element regeneration or re-import.

Optional additional elements include: multi-builder operation where the same brand system is applied across distinct sites using different builders; webhook-driven automatic propagation of brand-system changes to subscribed sites; edit-detection logic that preserves customer manual modifications when re-synchronizing; AI-driven content population within the adapter output; and agency-tier multi-site orchestration.

5. Detailed description

5.1 Brand-system input format

The invention receives a brand system expressed as a structured collection of design tokens. The brand system may be authored by the customer directly in the invention's companion dashboard, imported from a DTCG-compatible token file, extracted from a design tool (Figma, Sketch), or generated by AI from a brand brief. The brand system is internally stored in a canonical form independent of any specific target builder.

In the primary embodiment, the brand system comprises:

  • Color tokens: primary (brand), secondary (accent), neutral (text/background scale), semantic (success/warning/error/info), typically expressed as N-step scales (e.g., 12-step Radix-style palette)
  • Typography tokens: font family, size scale (fluid: min/max viewport + min/max size + scale ratio), weight, line height, letter spacing
  • Spacing tokens: size scale (fluid) for gaps, padding, margin
  • Radius tokens: small, medium, large, pill, atomic (button-specific)
  • Layout tokens: container max-width, breakpoints (mobile, tablet, desktop), section padding scales
  • Voice tokens: brand personality descriptors that influence AI-generated copy (formal/casual, technical/approachable, etc.)

5.2 Target-builder configuration input

The invention accepts, as a second input, a representation of the customer's website-builder configuration. This representation may be obtained by one of several mechanisms:

  • Customer uploads an exported configuration file via the invention's dashboard (e.g., Breakdance Settings Export, Bricks Templates JSON)
  • A companion plugin installed on the customer's CMS reads current configuration via WordPress options tables or post-meta and transmits to the invention's backend over authenticated REST
  • Customer authorizes OAuth access to a platform API (e.g., Shopify Admin API) and the invention retrieves theme configuration directly
  • Empty configuration (“greenfield” site) where the invention generates from scratch rather than applying to existing content

5.3 Builder-specific adapter architecture

A library of builder-specific adapters is maintained, one per supported website-builder. Each adapter is a software module that encapsulates the target builder's idiosyncratic characteristics:

  • Element schema: the set of valid element types (e.g., Section, Grid, Container, Heading, Text, Button, Image) and their property namespaces
  • Nesting model: whether the builder uses nested trees (Breakdance) or flat arrays with id-based parent/child references (Bricks)
  • CSS-variable binding: how element properties can reference design tokens (e.g., Bricks _background.color = "var(--brandrai-brand)"; Breakdance design.container.background.color = "var(--hcl-brand-9)")
  • Responsive encoding: how breakpoint-specific overrides are stored (e.g., Breakdance's breakpoint_base/breakpoint_tablet_portrait map vs. Bricks' :tablet/:mobile suffixes)
  • Import format: the JSON shape, file extension, or API payload schema the builder accepts for re-import
  • Global-style injection point: where the brand tokens themselves are written (e.g., bricks_theme_styles WP option for Bricks; Breakdance global-settings export for Breakdance)
  • CSS-regeneration trigger: how the builder rebuilds generated CSS when globals change (e.g., wp bricks regenerate_assets CLI command, or file-system cache invalidation)

A dispatcher component selects the correct adapter based on the identified target builder. Identification may be automatic (plugin inspects active plugins on the WP site; Shopify OAuth scope reveals theme capabilities) or explicit (customer selects from a dropdown).

5.4 Token-aware page-content generation

The invention maintains, separately from adapters, a library of page templates expressed in a builder-agnostic intermediate representation. Templates describe page structure in abstract terms (“Section → Grid(4 cols) → Card × 4 → Heading + Text + Button”) without reference to any specific builder's schema. Templates are parameterized by content: copy text, images, link targets, specific token references.

To generate a concrete page for a target builder:

  1. Select an abstract page template (“Pricing,” “Homepage,” “Industry Landing,” etc.)
  2. Apply user-specific content (pricing tiers, industry name, product list) by substituting parameters
  3. Pass the parameterized abstract template through the selected builder-specific adapter
  4. Adapter emits concrete elements in the builder's native format, binding each element's styled properties to the builder's native CSS-variable references (as opposed to hard-coded hex values, pixel dimensions, or inline styles)
  5. Adapter wraps the result in the builder's expected outer envelope (e.g., {"source": "...", "element": {...}} for Breakdance; {"content": [...], "version": "...", "name": "..."} for Bricks)
  6. Output payload is returned to the customer's plugin, or delivered via webhook / direct API push to the target site

The inventive element here is the intermediate-representation + adapter architecture: one set of page templates supports N target builders by virtue of the adapter library being extensible. Adding a new builder (e.g., Oxygen, Kadence, Divi) requires writing a new adapter only — existing page templates are automatically available on the new builder.

5.5 Automatic cascade mechanism

Because generated elements reference brand tokens through the target builder's native CSS-variable infrastructure (rather than hard-coded values), any change to the brand system that updates those variables propagates to all generated content automatically when the builder re-renders or regenerates CSS.

In operation:

  1. Customer changes brand color (e.g., emerald → teal) in the invention's dashboard
  2. Dashboard pushes updated brand-system payload to each connected site's plugin via webhook
  3. Plugin writes updated CSS-variable definitions to the builder's global-style storage (bricks_theme_styles, Breakdance global settings, Shopify theme settings)
  4. Plugin triggers the builder's CSS-regeneration mechanism
  5. Every page on the site (however many were generated) now renders in the new color — no page regeneration needed

The invention is distinguishable from prior art that requires regeneration of each page's content to update styling: the cascade mechanism is instant and scales to N pages without N regeneration operations.

5.6 Edit-detection and round-trip awareness

After initial generation, customers typically customize pages manually within their builder. The invention includes edit-detection logic that preserves those manual changes when subsequent brand-cascade or page-update events occur:

  • At generation time, the invention records a hash of the generated payload per-page
  • Prior to any overwrite, the invention fetches the current page state from the builder and compares its hash to the last-known generated hash
  • If hashes match: safe to overwrite
  • If hashes differ: customer has edited; the invention refuses overwrite by default and returns a conflict status to the dashboard, where the user can review the diff and choose to (a) preserve their edits and accept only brand-cascade changes, (b) overwrite their edits with the new generated content, or (c) cancel
  • Advanced embodiments perform element-level diff reconciliation: keep customer-edited elements, replace un-edited elements with newly-generated content

5.7 Companion plugin / connector

For WordPress-based builders, the invention operates in conjunction with a companion plugin installed on the customer's site. The plugin:

  • Authenticates to the invention's backend via OAuth 2.0
  • Exposes REST endpoints for brand-cascade and page-upsert operations, authenticated by HMAC signatures with shared secret
  • Performs Bricks/Breakdance/Elementor-specific writes to WP options and post-meta
  • Triggers builder-native regeneration commands where applicable (wp bricks regenerate_assets)
  • Logs sync events to a local audit table for debugging and customer visibility
  • Detects which builder is active and gracefully refuses unsupported builders

For Shopify, the companion is either a Shopify App (using Admin API) or deployment via Shopify's GitHub theme integration. No WordPress equivalent exists for static-site-generator or headless-CMS embodiments; those rely on direct API or git-commit-based deployment.

5.8 Multi-site and agency orchestration

An agency or multi-brand operator may connect N independent sites to a single invention account. Each site maintains its own authentication token and plugin instance, but all share access to a common brand-system library (the agency's client portfolio). Changes to a specific client's brand system propagate only to that client's connected sites. Generation operations can be batched across multiple sites simultaneously.

5.9 AI-driven content population

In enhanced embodiments, the invention does not merely generate structure; it generates content within that structure using large language models conditioned on: the brand's voice tokens, the target page's vertical context (cannabis, supplement, luxury, etc.), and the specific page template's purpose (hero copy, feature descriptions, FAQ questions). Generated copy is inserted into the abstract template's content slots before adapter emission.

A library of vertical-specific prompt templates parallels the page-template library: a “Pricing” page generated for a cannabis brand uses cannabis-vertical copy prompts; the same template generated for a supplements brand uses supplement-vertical copy prompts. The AI provider is pluggable (GPT, Claude, Gemini, open-source models).

6. Representative embodiments

The method and system may be embodied across a diverse set of website-builder targets. Each target becomes a distinct dependent claim. Known embodiments at the time of filing include:

6.1 — WordPress page-builder adapters

  • Breakdance Builder (Soflyy): nested element tree, EssentialElements\\* type names, breakpoint_base responsive encoding; global settings via export/import JSON
  • Bricks Builder: flat element array with id/parent/children relationships, native CSS-variable architecture via bricks_theme_styles option, wp bricks regenerate_assets CLI command
  • Elementor: widget-based nested model, stores in _elementor_data post meta, CSS globals via elementor_kit_settings
  • Oxygen Builder (Soflyy, predecessor to Breakdance): similar schema to Breakdance with minor differences; shared adapter possible
  • Kadence / Kadence Blocks: Gutenberg-based blocks with custom attributes; design system via theme.json
  • Gutenberg core blocks: block.json schemas per block type, theme.json for globals, post_content stores rendered block markup
  • Divi (Elegant Themes): shortcode-heavy format with proprietary modules; more complex adapter
  • Beaver Builder: similar widget-model to Elementor, different storage format
  • GeneratePress Elements + Blocksy: theme-native customization systems

6.2 — Shopify embodiments

  • Shopify Theme path: a BrandrAI-designed Shopify theme that exposes design tokens as theme settings (config/settings_schema.json); brand system is applied by updating config/settings_data.json via GitHub integration or theme API; deploys through Shopify's GitHub theme integration for version-controlled push
  • Shopify App path: OAuth app reads/writes theme settings, metafields, and section definitions via the Admin API; operates against customer's existing theme of choice; pushes brand tokens into the theme's settings schema
  • Shopify Online Store 2.0 sections: native section JSON schema supports custom settings; invention generates section definitions with token-bound settings

6.3 — Headless CMS embodiments

  • Sanity, Contentful, Strapi, Prismic: adapter writes brand tokens to the CMS's global settings / design-system content model; generates page documents matching the CMS's schema
  • Storyblok, Hygraph, Directus: similar to above

6.4 — Static site generator embodiments

  • Next.js with Tailwind or CSS-variable-based design system: adapter emits page components and/or MDX files; brand tokens written to CSS-variable definitions in globals
  • Astro: similar, outputs .astro components and design-token files
  • Gatsby: outputs .jsx pages + theme-ui or similar design-token provider
  • Hugo / Jekyll / Eleventy: template and data-file outputs appropriate to the generator
  • In static-site embodiments, “deploy” occurs via git commit to a repository linked to Vercel, Netlify, Cloudflare Pages, or equivalent hosting; the invention may use the GitHub/GitLab API to create pull requests or direct commits

6.5 — Additional embodiments

  • Webflow: Webflow-specific schema via CMS API; possible via their Designer API in future
  • Framer: Framer CMS API writes
  • Squarespace: more limited API access; adapter writes via their Developer Platform where permitted
  • Mobile app builders: adapter emits component definitions for Flutter, React Native, or native iOS/Android UI frameworks
  • Email template builders (Mailchimp, Klaviyo, Customer.io): adapter emits MJML or platform-native email templates with brand-bound styling
  • Presentation tools (Google Slides, PowerPoint templates): adapter emits deck templates with brand-bound theming
  • Chat / conversation interfaces: adapter emits customer-facing chat UI themes

6.6 — Vertical-specific content embodiments

The page-template library may include vertical-specific variations, each constituting a dependent claim:

  • Cannabis industry pages (hero, lab-results, dispensary-locator, age-gate, compliance-disclaimer)
  • Regulated-product pages (CBD, kratom, supplement, peptide, mushroom verticals)
  • Ecommerce pages (product-detail, cart, checkout, return-policy)
  • Service-business pages (booking, service-list, testimonials, case-studies)
  • Content-site pages (blog-index, article, author-bio, newsletter-signup)
  • Luxury / premium-brand pages (curated-collection, brand-story, atelier-visit-booking)

6.7 — Integration with authenticity-attestation pipeline

In combined embodiments, generated pages include verification-bound elements that surface outputs from a companion authenticity-attestation system (e.g., COA verification, product-certificate verification):

  • Generated product-detail page includes an embedded “Verify” widget that displays the current verification status of the product
  • Generated compliance page references live attestation records and displays QR codes that link to public verification pages
  • Brand-system changes cascade through both page styling and verification-widget theming simultaneously

7. What is novel and non-obvious

The inventive combination rests on elements individually known but never combined in the manner claimed:

  1. Builder-agnostic intermediate representation with builder-specific adapter dispatch. Prior-art page-generation tools are tightly coupled to a single target platform. The adapter-library architecture, where one abstract template library produces valid output for N distinct website builders through pluggable adapter modules, is novel in this domain.
  2. CSS-variable-native token binding during generation. Prior art either produces pages with hard-coded style values (losing cascade) or generates CSS-variables into source code (not applicable to third-party page builders' existing infrastructure). Producing page content that references the target builder's own CSS-variable infrastructure is novel.
  3. Automatic cascade without page regeneration. Prior systems require regeneration of each page to propagate brand changes. The cascade-via-shared-variable approach is novel in applying design-token-propagation semantics to third-party builder outputs.
  4. Edit-detection with hash-based reconciliation. The round-trip model (detect customer edits, refuse overwrite without consent, optionally element-level diff reconciliation) applied to page-builder content is novel.
  5. Multi-site cascade from a single brand-system source. Agencies and multi-brand operators pushing brand-system updates to N heterogeneously-configured sites from a single control point is novel.
  6. Integration with authenticity-attestation layer for cross-cascading. Generated pages that include live-attestation widgets themed via the shared brand-cascade is a novel integration with the authenticity-verification domain.
  7. Vertical-specific page-template library with domain-conditional AI copy population. Combining industry-specific abstract templates with vertical-specific AI copy generation during adapter emission is novel.

Counsel should draft independent method and system claims covering the combination of: (a) brand-system input, (b) builder-specific adapter selection from a library, (c) token-aware generation, (d) import-compatible output, (e) automatic cascade mechanism. Dependent claims should address each enumerated builder, each vertical, each platform target (WP/Shopify/headless/SSG), edit-detection, multi-site orchestration, AI copy integration, and authenticity-layer integration.

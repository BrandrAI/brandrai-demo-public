2 · Usage meters (Stripe Billing)

Two independent meters, two overage pricing models.

COA verifications meter
  • Event name: coa_verification
  • Aggregation: sum (each verify = 1 unit)
  • Billing: metered, aggregated over the billing period
  • Included allowance enforced in-app (not Stripe); overages reported to meter
  • Overage price: $2.00 per unit, post-threshold
  • Price type: graduated — tier 1 (0 to included) = $0, tier 2 (included+) = $2.00
Studio credits meter
  • Event name: studio_credit
  • Aggregation: sum (variable cost per action: 1-20cr)
  • Billing: not metered on subscription — overage handled via pack purchases
  • When balance hits 0, UI blocks generation and shows pack picker
  • Pack purchases add to a separate studio_credit_balance field that persists across months (packs never expire)

Design decision: COA overage bills at $2/unit because authentication is a recurring per-unit need (compliance-driven). Studio overage uses packs because generation is bursty (campaign cycles) and packs give users control — they hate surprise bills for creative work. This hybrid is intentional.

4 · Early-adopter coupon

One coupon locks discounted rates through 2026.

Coupon config
id: EARLY_ADOPTER_2026
name: "Early adopter — locked through 2026"
duration: repeating
duration_in_months: [ months remaining in 2026 at time of signup ]
percent_off: ~50.25% (Starter), 33.44% (Growth), 31.29% (Business)
applies_to: all 3 paid subscription products
max_redemptions: 500 (tune as needed)
redeem_by: 2026-12-31T23:59:59Z

Implementation note: the cleanest approach is 3 separate coupons (one per tier) each with a specific fixed amount_off that brings the price to the early-adopter rate:

TierFull priceDiscounted priceCoupon amount_off
Starter$99/mo$49/mo$50.00 (monthly, repeating)
Growth$299/mo$199/mo$100.00 (monthly, repeating)
Business$799/mo$549/mo$250.00 (monthly, repeating)

Duration semantics: set duration: repeating + duration_in_months = calendar months remaining until Dec 31 2026 at signup. Simplest path: hardcode to 12 months and accept that a Sept 2026 signup locks through Aug 2027 — close enough for marketing purposes. Don't over-engineer.

5 · Webhook events to handle

Minimum set for launch.

EventWhen it firesWhat to do
checkout.session.completedUser finishes checkout (sub or pack)Create subscription record OR increment studio_credit_balance based on mode
customer.subscription.createdNew sub startsSet plan_tier on user, apply included_coa_verifications + included_studio_credits to balance, reset monthly counters
customer.subscription.updatedTier change, pause, resumeUpdate plan_tier, prorate allowances if upgrading
customer.subscription.deletedCancelled at period endDowngrade to free tier at period end; do NOT clear studio_credit_balance (packs persist)
invoice.payment_succeededMonthly billing cycle successfulReset monthly meters (included COA verifications, included Studio credits)
invoice.payment_failedCard declinedEmail user; put account in grace period; do not reset meters

Signature verification: every webhook MUST verify the Stripe signature (stripe-signature header + webhook secret). Use Laravel's StripeWebhookController or equivalent. No unsigned webhooks in prod.

7 · MVP vs v1.1 scope

Ship the minimum; iterate post-launch.

MVP (by Apr 28)
  • 4 subscription products (Free, Starter, Growth, Business)
  • Monthly + annual prices
  • 3 early-adopter coupons
  • checkout.session.completed + customer.subscription.* webhooks
  • Plan entitlement config in app
  • Test mode end-to-end tested before live switch
v1.1 (post-May 1)
  • 4 Studio credit packs + checkout flow
  • COA overage metering ($2/unit) via Stripe Billing meters
  • Credit pack UI in-app (pack picker modal from Studio UX spec §8)
  • Usage dashboard (Studio credit + COA meter history)
  • Plan change / upgrade flow
  • Customer portal for self-serve billing

Why this split: at May 1 the first 10-50 operators are mostly Free + Starter signups testing the product. Overage metering and pack purchases won't matter for 2-4 weeks. Ship subscription + checkout first; everything else is post-launch optimization.

8 · QA checklist before go-live

Run these before flipping the switch.

  1. All 4 products exist in Stripe with correct names and descriptions
  2. All 4 products have both monthly + annual prices (where applicable)
  3. Price IDs copied into config/brandrai_plans.php
  4. 3 early-adopter coupons created + tested on signup flow
  5. Webhook endpoint configured in Stripe (/webhooks/stripe) with signing secret
  6. Webhook secret added to .env as STRIPE_WEBHOOK_SECRET
  7. Test signup to Starter tier completes, user record shows correct plan_tier
  8. Test signup to Growth tier with early-adopter coupon shows $199/mo (not $299)
  9. Test card (4242 4242 4242 4242) successful payment flow
  10. Test card failure flow (4000 0000 0000 9995 — always declines)
  11. Upgrade from Starter to Growth prorates correctly
  12. Cancellation flow completes; user downgrades to free at period end
  13. Invoice emails going out from Stripe with correct branding
  14. Tax (if needed) configured via Stripe Tax or marked exempt
  15. Live mode keys swapped in at cutover; test mode keys left in staging

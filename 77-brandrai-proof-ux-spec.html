1 · How Proof lives inside BrandrAI

Same app, same login. New feature, not new product.

Proof is not a separate app or separate login. It's a new top-level workspace feature inside app.brandrai.com that any logged-in user can access. Architecturally it reuses 90% of what's already built for COA verification:

Shared with COA verification New for Proof
✓ Auth + workspace (Sanctum) Generic content upload (not just COA)
✓ SHA-256 hashing (FileService) Domain-type dropdown (patent / academic / creative / legal / etc.)
✓ Public UUID verify page Generic “proof” verify-page template (not COA-specific)
✓ Queue + worker (ProcessProofJob mirrors ProcessCoaVerificationJob) Domain-conditional extraction prompt library
✓ AI service (AiService) Generic metadata schemas per domain
✓ Pusher broadcasting Signed PDF certificate generator (new — CertificateService)
✓ QR generation (qrcode) Trusted timestamp source (RFC 3161 TSA or equivalent)
✓ Pricing + credits system Optional on-chain anchoring (roadmap Q4 2026)

The fundamental engineering lift is much smaller than building a new product. We're exposing the existing engine with a broader upload flow and a generic verify page, plus adding two net-new components (signed certificate, trusted timestamp source).

3 · Core user flow

Four steps, under 60 seconds.

1

Upload

Drag-and-drop any file. PDF, image, video, dataset, code archive. Up to 100MB (paid tier limits).

2

Declare domain

Select preset: Legal / Patent / Academic / Creative / Medical / Other. Optional description + tags.

3

Processing (async)

Status bar: hashed → timestamped → AI analyzed → certified. Live updates via Pusher. 15-45s typical.

4

Certificate + share

Download signed PDF. Copy public verify URL. Download QR code. Email / share directly.

The entire flow mirrors the existing COA verification wizard pattern (Verifications/Create.tsx) so the learning curve for existing users is near-zero. The only novel step is the domain-preset selector.

4.3 Detail view (/proof/{verify_id})

Merchant-side authenticated page. Shows everything:

  • Verdict badge + confidence + risk level
  • Full SHA-256 hash in copy-friendly format
  • Trusted UTC timestamp with source attribution
  • AI-extracted metadata (structured table by domain)
  • Findings and red flags (if any)
  • Original file preview (PDF / image viewer inline)
  • Actions: Download Certificate PDF · Copy public URL · Download QR · Revoke

Consumer / third-party view. No auth. Mobile-first. Shows:

5 · The signed certificate

This is the deliverable. Get it right.

The certificate PDF is what users show to a patent examiner, opposing counsel, an FDA auditor, or a judge. Its quality defines the product's perceived legal defensibility. Not an afterthought.

Certificate contents (single page, ~8.5×11" or A4):

  1. Letterhead: “BrandrAI, Inc. — Certificate of Cryptographic Attestation”
  2. Unique certificate ID (verify_id + random suffix, for revocation tracking)
  3. Attested content summary: filename, size, declared title, domain
  4. Cryptographic hash (SHA-256, full 64-hex-char, monospace)
  5. UTC timestamp + source attribution (“Timestamp from RFC 3161 TSA at provider-name”)
  6. AI-extracted metadata summary (2-4 key fields from domain schema)
  7. Verdict + confidence + risk level
  8. Public verification URL + embedded QR code
  9. Methodology statement (~3 sentences on how attestation works)
  10. “Self-authentication foundation” boilerplate referencing FRE 902(13) and 902(14)
  11. Digital signature block — the certificate itself is signed by BrandrAI, Inc.'s signing key
  12. Optional on-chain anchoring reference (block number + transaction hash for enterprise tier)

Generation technology: we render the certificate with Dompdf or wkhtmltopdf from an HTML template (resources/views/proof/certificate.blade.php), then sign the resulting PDF with setasign/fpdi + setasign/fpdf_tpl (or equivalent) using a software signing certificate. Annual certificate renewal.

For enterprise tier: optional notarization add-on where a licensed notary witnesses the certificate issuance via e-notarization (RON — Remote Online Notarization). Partnership opportunity with Notarize.com or equivalent.

6 · The cryptographic layer

What production-grade actually means.

Everything above the certificate is product work. The cryptographic layer is what makes the product defensible. Five distinct pieces have to be solved:

1

Hashing

Current state: ✓ done. FileService computes SHA-256 via streaming hash_file().

Production hardening needed:

  • Algorithm abstraction layer — record the algorithm in the DB so it can be swapped later (SHA-3, BLAKE3, post-quantum)
  • Client-side hashing option (Web Crypto API in browser) to avoid uploading the file at all for privacy-critical use cases — HIPAA-ready architecture
Effort: ~3 days
2

Trusted timestamping

Current state: we use now() from app server clock. Not acceptable for legal defensibility.

What's needed:

  • Integrate RFC 3161 time-stamping authority (TSA) — paid services include FreeTSA (free but lesser legal weight), DigiCert TSA (~$500/yr), Sectigo TSA, or GlobalSign TSA
  • Store the TSA response as a binary blob alongside the verification record
  • Expose the TSA signature on the public verify page and in the certificate
  • Fallback handling if TSA is unreachable (queue, retry, or secondary TSA)
Effort: ~1-2 weeks including TSA vendor selection, contract, integration, testing
3

Certificate PDF signing

Current state: not built. No signed PDF today.

What's needed:

  • Obtain a code-signing / document-signing certificate for BrandrAI, Inc. from Sectigo or DigiCert (~$200-500/yr)
  • Store the signing key in HashiCorp Vault or equivalent secrets manager
  • Implement CertificateService::generate() using Dompdf + SetaPDF or TCPDF + OpenSSL
  • Support LTV (Long-Term Validation) — embed timestamping signature so the PDF remains verifiable even after the signing cert expires
Effort: ~2 weeks including vendor paperwork
4

Public re-verification UI

Current state: verify page shows hash but doesn't re-compute or compare.

What's needed:

  • Client-side hash computation via Web Crypto API (crypto.subtle.digest) in the verify page
  • Drag-and-drop area: “Drop your copy of this document here to re-verify”
  • Compare computed hash to stored hash — green check or red mismatch
  • Never upload the re-verification file to the server — purely client-side so the user retains privacy
Effort: ~1 week
5

On-chain anchoring (optional, Q4 2026)

Current state: not built. On the Four Circles roadmap.

What's needed:

  • Daily batch job that builds a Merkle tree of all verifications completed in the prior 24 hours
  • Publish the Merkle root as an OP_RETURN transaction on Bitcoin (~$2-20/tx depending on mempool)
  • Store the transaction hash, block height, and Merkle proof per verification
  • Update verify page + certificate to surface the on-chain anchor when available
  • For higher-frequency tier: use OpenTimestamps calendar servers instead of direct Bitcoin writes
Effort: ~2-3 weeks (deferred to Q4 2026)

8 · The Metrc sidebar

The easy part, confirmed easy.

You mentioned a friend who can provide Metrc keys if Metrc's response stalls. Excellent backup — use it.

What already works today:

  • ✓ MetrcService with state-specific URL routing (OH, CO, OR, MI + expandable)
  • ✓ METRC_DEMO_MODE=true returns realistic stub data for dev/demo without live keys
  • ✓ METRC_VENDOR_KEY + METRC_USER_KEY env-var wiring
  • ✓ Package-tag lookup endpoint + lab-test results endpoint integrated
  • ✓ Normalization to the unified PackageVerification model
  • ✓ UI toggle between Metrc primary and Upload COA fallback

What happens when keys arrive:

  1. Add the keys to production .env
  2. Flip METRC_DEMO_MODE=false
  3. Run a single sandbox verification to confirm connectivity
  4. Ohio package tags resolve live; verification page shows METRC-ANCHORED · LIVE PULL badge

From friend's keys vs official Metrc keys: functionally identical for demo / prelaunch purposes. For a production partnership with the state regulator visible at the Ohio launch, official Metrc-issued keys carry more weight. For testing + demo, the friend's keys let us be unblocked today.

Note: using a friend's keys means queries are attributed to their Metrc account. This is fine for short-term testing; for actual customer-facing production it creates an audit attribution issue that both you and the friend would prefer to avoid. Target having our own keys before the May 1 public event.

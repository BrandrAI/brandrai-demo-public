1 · Product thesis

DocuSign stores. BrandrAI proves.

DocuSign / Dropbox Sign / PandaDoc
  • Signature is an image + typed name
  • Audit trail lives on DocuSign's servers — trust their word
  • No cryptographic proof of document integrity post-signing
  • Cannot independently re-verify without the vendor's API
  • $40-60/user/month, seat-based
BrandrAI Sign
  • Signature is cryptographically bound to the document via SHA-256 hash
  • Audit trail is publicly re-verifiable at /sign/{uuid}
  • Trusted timestamp + optional on-chain anchoring
  • Any third party can validate the signature without BrandrAI's cooperation — the hash speaks for itself
  • Included in paid tiers + priced per-signature via Studio credits

Positioning statement: "Your signature, cryptographically verifiable. If a court, counterparty, or regulator asks whether a signed document was altered after signing, you can prove it in seconds — not beg a vendor to authenticate for you." This is a differentiator, not a feature parity play.

Target users: cannabis operators signing lab service agreements, distributor contracts, retailer MSAs, employee NDAs/CIIAs, investor SAFEs — audiences where "was this document altered?" is a non-rhetorical question.
Competitive edge: only tamper-evident e-sign platform with public cryptographic re-verification. Patent 75 (Proof primitive) gives us a defensibility moat.
Revenue story: replaces ~$600/year DocuSign spend per user. Included in BrandrAI subscription. Retention driver — customers don't churn if their contracts are in your system.

3 · Core user flow

5 stages, end-to-end.

Stage 1
Create agreement
Upload PDF or use template. System hashes document on creation.
Stage 2
Configure fields + signers
Drop signature blocks, initial blocks, date fields. Assign to named signers.
Stage 3
Route for signing
Serial or parallel. Email notifications with tokenized signing links.
Stage 4
Sign
Signer auths, consents, places signature. Server captures IP, device, timestamp, IAB consent.
Stage 5
Certificate + verify
Final PDF + signing certificate + public verify page at /sign/{uuid}.

Key architectural decision: the signed document is a new artifact — not the original PDF with a stamp. The system generates a final PDF that embeds a signature block per signer + an appended certificate page showing the hash, signers, timestamps, and the public verify URL. The original, unsigned PDF is preserved separately. Both hashes are recorded.

5 · Data model

Two new tables. Reuse everything else.

-- agreements (the "envelope")
agreements
  id                   UUID PK
  workspace_id         FK
  title                str           -- human-readable name
  original_file_hash   str(64)       -- SHA-256 of uploaded PDF
  original_file_path   str           -- storage path
  final_file_hash      str(64)       -- SHA-256 of signed PDF (set on completion)
  final_file_path      str           -- storage path of signed PDF
  verify_id            UUID          -- public-facing UUID (encoded in QR)
  status               enum          -- draft | out_for_signature | completed | voided | expired
  routing_mode         enum          -- parallel | sequential
  template_id          FK nullable   -- if created from template
  created_by_user_id   FK
  created_at           timestamp
  expires_at           timestamp     -- default 30 days
  completed_at         timestamp nullable
  metadata             json          -- tags, custom fields

-- signers (one row per required signer on an agreement)
signers
  id                   UUID PK
  agreement_id         FK
  signer_name          str
  signer_email         str
  signing_order        int           -- for sequential routing
  sign_token           str           -- secure random token in signing URL
  token_expires_at     timestamp
  status               enum          -- pending | signed | declined | expired
  signed_at            timestamp nullable
  signed_ip            str           -- IPv4/IPv6
  signed_user_agent    str
  signed_geolocation   str nullable  -- from IP (optional)
  signature_image_path str nullable  -- drawn sig image
  typed_signature_text str nullable  -- typed sig
  consent_acknowledged boolean       -- ESIGN consent checkbox
  auth_method          enum          -- email_link | sms_otp | sso | id_verify

-- field_placements (where signature/initial/date fields appear on the PDF)
field_placements
  id                   UUID PK
  agreement_id         FK
  signer_id            FK
  page_number          int
  x, y, width, height  int           -- PDF coordinates
  field_type           enum          -- signature | initial | date | text | checkbox
  filled_value         str nullable  -- captured value on sign
  filled_at            timestamp nullable

What we reuse from the existing COA Verify schema: files table (storage + hashing), users/workspaces (tenant isolation), notifications (Pusher broadcasts), queue jobs (async PDF generation + certificate rendering). Most of the plumbing already exists.

Why both hashes are included: the original hash proves what the document looked like before signing (what signers agreed to). The final signed hash proves the completed document with all signatures. Comparing both lets any party validate that no post-signing tampering occurred.

7 · Public verify page for signed docs

Same pattern as COA verify — no auth required.

└─ brandrai.com/sign/ec7f82a1-.... ───────────────────
│                                                         │
│  ✅ DOCUMENT SIGNED AND VERIFIED                       │
│                                                         │
│  Ambassador Agreement                                    │
│  Completed 2026-07-16 09:47:51 UTC                       │
│                                                         │
│  Signed hash:  8e71...d3c0                               │
│  [ Re-verify against your copy ]                         │
│    └ drag + drop your PDF to compute hash locally      │
│                                                         │
│  Signers                                                 │
│  ✓ Max Juhasz (BrandrAI)           07/15 14:25:11 UTC │
│  ✓ Krysta Jones (THC Girls)        07/16 09:47:51 UTC │
│                                                         │
│  [ Download signed PDF ]   [ Download certificate ]      │
│                                                         │
│  Powered by BrandrAI Proof · cryptographically bound.   │
&boxvur;──────────────────────────────────────────────────────

The killer feature: re-verification. Anyone with a copy of the signed PDF can drag-and-drop it onto the verify page. The client computes SHA-256 locally and compares against the stored hash. Match = authentic. Mismatch = altered. Zero trust in BrandrAI required — the math either works or it doesn't.

Privacy consideration: the verify page shows signer names and roles but not the signed document contents (unless user enables public visibility). Default is "hash + metadata only"; signers can opt to make the full PDF publicly accessible.

8 · Legal compliance

ESIGN, UETA, eIDAS, and international signer frameworks.

FrameworkJurisdictionOur compliance approach
ESIGN Act (2000)US federalIntent + consent + association + retention. Captured via consent checkbox, server-side audit trail, immutable certificate.
UETAUS state (48 states)Mirrors ESIGN at state level. Our implementation satisfies both concurrently.
eIDAS (Reg 910/2014)EUOur v1 ships as a "simple electronic signature" (SES) — legally valid for most commercial contracts. "Advanced" (AES) requires qualified trust service provider integration — v2 roadmap item.
eIDAS (UK)United KingdomPost-Brexit UK eIDAS regime is nearly identical to EU eIDAS. Our SES implementation satisfies both.
PIPEDA + e-sig guidanceCanadaCanada's PIPEDA + provincial e-sig acts accept similar intent/consent/retention model.
Electronic Transactions Law (2023)VietnamVietnam updated its e-sig law in 2023 to recognize electronic signatures with proper authentication. Relevant for Dang as co-founder.
Indian IT Act (2000, amended)IndiaIndia distinguishes between "electronic signature" (our SES) and "digital signature" (DSC via govt CA). SES is valid for most commercial contracts; high-value real estate/gov contracts require DSC.
Document types we explicitly DO NOT support at v1

Wills, codicils, trusts, adoptions, family law filings, court orders, some UCC commercial paper transfers, notary-required real estate transactions. These require specialized processes (witnesses, notarization, court filing) that are out of scope. UI shows a warning if user attempts to upload a document type matching these patterns.

Compliance-as-code: the consent checkbox, intent language, IP capture, timestamp capture, and certificate generation are all required steps in the signing state machine. You cannot complete a signature without satisfying ESIGN's four pillars. Compliance is enforced at the code level, not as a post-hoc review.

Example costs:

  • 2-party NDA with basic email auth: 2cr + 1cr = 3cr (~$0.75 at Creator pack rate)
  • 3-party distribution agreement with SMS OTP: 2cr + 1cr + 1cr + 3x1cr SMS = 8cr (~$2)
  • High-stakes vendor MSA with on-chain anchoring + ID verify: 2cr + 3cr + 5cr = 10cr (~$2.50)

Pricing vs. DocuSign: DocuSign Business Pro is ~$60/user/month with unlimited envelopes. BrandrAI Sign at 2cr/envelope comes to ~$0.50 per envelope. A user sending 20 contracts/month pays ~$10 in Studio credits vs. $60 DocuSign. For heavy users (50+ envelopes/month), DocuSign is cheaper — but they're paying for UI/UX, not cryptographic verification. The BrandrAI pitch is not "cheaper" — it's "verifiable."

10 · Patent coverage mapping

How Sign lands under patent 75 (Proof primitive).

BrandrAI Sign is explicitly covered by the Proof patent disclosure (75) and its expanded embodiments (§8B). Key sections of 75 that Sign implements:

Patent 75 sectionHow Sign implements it
§5.2 Cryptographic hashingSHA-256 of original + final signed PDF
§5.3 Trusted timestamp bindingServer-side RFC 3161 timestamp on each signer event
§5.4 Domain-conditional AI extractionOptional — auto-extract field types, counterparties, effective date from PDF (roadmap v1.1)
§5.8 Public verification pageDirect implementation at /sign/{uuid}
§5.9 Signed attestation certificateCertificate of Completion page appended to final PDF
§5.11 On-chain anchoring (optional)Premium option — batched hash anchoring to public blockchain
§8B.4 Legal + evidentiary attestationSign is this embodiment — explicitly listed as a dependent claim in the disclosure
§8B.8 Multi-party attestationMulti-signer agreements map 1:1 to this embodiment

IP strategy implication: filing patent 75 before May 1 (as Edison is executing) directly protects BrandrAI Sign as a product line. Any competitor building tamper-evident e-sign on public verify pages would need to navigate around the 75 claims. This is one of the strongest commercial cases for filing 75 now rather than deferring.

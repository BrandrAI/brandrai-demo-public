2. Field of the invention

This invention relates to systems and methods for producing legally-meaningful attestations of digital content integrity and creation time. More specifically, it pertains to the combination of cryptographic hashing, trusted timestamping, large-language-model-driven metadata extraction, deterministic authenticity scoring, and public verification addressing into a single automated pipeline suitable for legal, regulatory, creative, academic, and commercial authenticity use cases.

3. Background — state of the art

Several partial solutions exist for digital document authentication but none combines all necessary elements into a user-accessible legally-meaningful attestation:

Blockchain timestamping (OpenTimestamps, Chainpoint, WordProof)Immutable hash + timestampNo metadata extraction, no certificate, no user-accessible verify page

No existing system combines (a) byte-level cryptographic hashing, (b) a trusted UTC timestamp bound to that hash, (c) LLM-based structured metadata extraction bound to the same hash, (d) deterministic forensic authenticity scoring of the extracted metadata, (e) a public UUID-addressed verification page that re-renders the hash for independent verification, and (f) an exportable signed attestation certificate suitable for evidentiary use — in a single workflow accessible to non-technical users.

The gap is particularly acute for use cases requiring rapid, cheap, legally-meaningful attestation of digital content creation time and integrity: pre-patent invention disclosures, academic research drafts, creative authorship claims, pharma adverse-event logs, compliance audit trails, contract version control, and court-admissible digital evidence chain-of-custody.

4. Brief summary of the invention

A computer-implemented method for producing a tamper-evident attestation record of arbitrary digital content, comprising:

  1. Receiving digital content from a user via an upload interface or API;
  2. Computing a cryptographic hash of the received content using a one-way hash function;
  3. Binding the hash to a trusted UTC timestamp obtained from an authoritative time source;
  4. Selecting a metadata-extraction prompt from a library of prompts based on a domain-type parameter supplied by the user or inferred from the content;
  5. Submitting the content to a large language model along with the selected extraction prompt, obtaining structured metadata output conforming to a predefined schema;
  6. Submitting the structured metadata to a deterministic forensic scoring function that evaluates the metadata against one or more authenticity heuristic axes, each scored on a bounded numeric scale;
  7. Calculating a categorical authenticity verdict from the axis scores via a fixed decision tree;
  8. Persisting the hash, timestamp, extracted metadata, scores, and verdict in a storage system indexed by a universally unique identifier;
  9. Generating a public verification page at a URL containing the unique identifier, wherein the page displays the verdict, extracted metadata summary, and the stored hash — and optionally re-computes the hash server-side as proof of continued integrity;
  10. Generating an exportable signed attestation certificate document containing the hash, timestamp, verdict, extraction summary, and verification URL.

Optional additional elements: anchoring the hash to a public blockchain; embedding the verification URL in a QR code; emitting real-time status events via publish-subscribe messaging during asynchronous processing.

5. Detailed description

5.1 Receiving content (step 1)

The invention receives arbitrary digital content from a user. The content may be (without limitation): a document (PDF, DOCX, TXT, Markdown), an image (JPG, PNG, HEIC, RAW), a video (MP4, MOV, WEBM), an audio file (MP3, WAV, FLAC), a dataset (CSV, JSON, Parquet), source code or archives (ZIP, TAR), or any binary file. Content is received via HTTP POST upload, drag-and-drop in a browser, programmatic API call with file payload, or chunked upload for files exceeding common size limits.

5.2 Cryptographic hashing (step 2)

The received content is processed through a cryptographic one-way hash function. In the primary embodiment, SHA-256 is used, computed in streaming fashion so that arbitrarily large content may be hashed without loading the entire payload into memory. The hash is deterministic — byte-identical content always produces the identical hash — and computationally infeasible to reverse or to produce collisions.

Alternative hash functions within the scope of the invention include: SHA-3 / SHA-384 / SHA-512; BLAKE3 for higher throughput; and post-quantum-resistant hashes such as SHAKE-256 or future NIST-standardized algorithms as cryptographic standards evolve. The system is hash-algorithm-agnostic; the method includes the step of recording which algorithm was used so that subsequent verification is unambiguous.

5.3 Trusted timestamp binding (step 3)

The hash is bound to a UTC timestamp obtained from a trusted time source. In the primary embodiment, the timestamp is obtained from an authoritative network time protocol (NTP) source maintained by the system operator, and the combined tuple (hash, timestamp) is recorded atomically in the storage system.

Alternative trusted-time embodiments within the scope include: RFC 3161 time-stamping authority (TSA) issued cryptographic timestamps; Roughtime protocol (Google's authenticated time protocol); blockchain block-time as the authoritative clock; hardware security module signed timestamps; and multi-source aggregated timestamps from independent authorities for higher trust guarantees.

5.4 Domain-conditional AI metadata extraction (steps 4-5)

A library of metadata-extraction prompts is maintained, each tuned for a specific domain: legal contract, clinical trial document, pharma adverse-event report, research manuscript, engineering specification, invention disclosure, creative work, photography, software release note, and generic fallback. Each prompt specifies the structured fields to extract and, optionally, domain-specific authenticity signals to observe.

The system selects the appropriate prompt based on (a) a domain parameter supplied explicitly by the user via an interface, or (b) an inferred domain determined by a preliminary classifier LLM call over the content, or (c) a default fallback prompt when no strong signal is available.

The content is submitted to a large language model via a chat-completion or vision-completion API, along with the selected prompt. The model returns structured JSON conforming to the prompt's schema. Example schemas include: for legal contracts — parties, effective date, governing law, clause summaries; for invention disclosures — inventor, filing date, technical field, novel elements; for research drafts — authors, institution, abstract, keywords, figures inventory.

The inventive step here is domain-conditional extraction: the same underlying LLM is reused across any document type, but the metadata schema extracted is tuned to the evidentiary value required by the specific use case.

5.5 Deterministic forensic scoring (step 6)

The extracted structured metadata is passed to a deterministic forensic scoring function, selected according to the same domain parameter. The scoring function evaluates the metadata against one or more bounded axes, where each axis represents an independent dimension of authenticity judgment.

In the primary embodiment, three orthogonal axes are used:

axis_1 = document_structure_score  (0-100, higher = better)
axis_2 = data_integrity_score       (0-100, higher = better)
axis_3 = red_flag_score             (0-100, higher = more red flags)

Each axis is computed by applying a set of rules to the extracted metadata. Rules may be hardcoded (e.g., “dates must be in chronological order”) or may themselves use an LLM with a specialized analysis prompt to evaluate qualitative signals (e.g., “are numerical values showing natural variance or suspiciously uniform?”). The rules are domain-specific and stored in a rules library that mirrors the prompt library in structure.

The inventive element is hybrid deterministic-stochastic scoring: the final scoring function is auditable and deterministic even though intermediate analyses may employ stochastic LLMs. This produces a reproducible, explainable authenticity verdict while still leveraging the flexibility of AI-based qualitative judgment.

5.6 Verdict calculation (step 7)

The axis scores are combined into a final categorical verdict via a fixed decision tree:

legitimacy = w1 * axis_1 + w2 * axis_2 + w3 * (100 - axis_3)
   where w1 + w2 + w3 = 1.0

if axis_3 >= threshold_red_flag_critical:
    verdict = FRAUDULENT
elif axis_1 >= threshold_good AND axis_2 >= threshold_good AND axis_3 <= threshold_red_flag_low:
    verdict = LEGITIMATE
elif axis_1 < threshold_weak AND axis_2 < threshold_weak:
    verdict = INCONCLUSIVE
else:
    verdict = SUSPICIOUS

confidence = legitimacy - (spread_between_axes * confidence_penalty_factor)

Weights (w1, w2, w3), thresholds, and confidence-penalty factor are parameters that may be tuned per domain or held constant across domains. The decision-tree structure itself is domain-invariant.

5.7 Persistence with UUID indexing (step 8)

The hash, timestamp, extracted metadata, axis scores, verdict, confidence, and domain parameter are persisted atomically to a storage system. A universally unique identifier (UUID, version 4 recommended) is generated and associated with the record; this UUID is the public-facing verification identifier.

The UUID is chosen so that it is infeasible to enumerate or guess other records (128-bit entropy). The storage system is typically a relational database with row-level access controls scoped to the submitting user's tenant (workspace). The original content itself is persisted separately to object storage, with the database row referencing its location.

5.8 Public verification page generation (step 9)

A public URL is constructed in the pattern /verify/{uuid} or equivalent, where {uuid} is the record identifier. This URL is accessible without authentication — any holder of the URL can access the verification page.

The verification page displays: the verdict with color-coded status badge; confidence and risk levels; a summary of extracted metadata (with domain-appropriate masking of sensitive fields); the binding UTC timestamp; the cryptographic hash rendered in a copy-friendly format; and a link or embedded viewer for the underlying content where privacy permits.

The inventive element here is third-party re-verification: the public page displays the stored hash, which any party in possession of a file claimed to be the attested content can re-hash using any standard SHA-256 implementation and compare. If the recomputed hash matches the displayed hash, the file is confirmed to be the attested content. If it differs, tampering or substitution has occurred. The verification is thus not dependent on trusting the system operator alone — any third party can independently confirm.

5.9 Signed attestation certificate (step 10)

On demand, the system generates an exportable signed certificate document (typically PDF) containing:

  • Full identity of the attesting system (BrandrAI, Inc.)
  • The UUID, UTC timestamp, and cryptographic hash with algorithm identifier
  • Summary of extracted metadata
  • Final verdict, confidence, and risk level
  • Public verification URL + embedded QR code
  • Digital signature applied by the system operator's signing key over the certificate content
  • Statement of methodology suitable for evidentiary foundation under FRE 902(13) or 902(14)

The certificate is designed to be usable as a self-authenticating record under the Federal Rules of Evidence, a supporting exhibit in court or administrative proceedings, or a compliance artifact for regulatory audit.

5.10 Optional QR code encoding

The public verification URL is optionally encoded into a QR code (ISO/IEC 18004 Model 2) using client-side or server-side generation libraries. The QR code enables a printed or on-pack distribution where a consumer with a standard smartphone camera can navigate to the verification page and inspect the attestation record without any specialized software.

5.11 Optional on-chain anchoring

For stronger third-party integrity guarantees, the cryptographic hash (or a Merkle root of a batch of hashes) may be anchored to a public blockchain by submitting it as a transaction or OP_RETURN output. Anchoring provides a second independent timestamp source that is infeasible to retroactively alter and can be verified by any party with blockchain access. Anchoring may be performed per-record or in batches to amortize transaction costs.

5.12 Optional real-time state broadcasting

When steps 4-7 are performed asynchronously (e.g., via a background queue worker), the system emits state-change events (pending → processing → completed | failed) on a publish-subscribe channel scoped to the submitting user's tenant. Client applications subscribing to this channel receive real-time status updates without polling. This enables responsive user interfaces during the asynchronous processing window.

7. What is novel and non-obvious

Each of the five constituent elements exists independently in prior art. The invention's novelty lies in their combination into a single pipeline, specifically:

  1. Combination of cryptographic hash + trusted timestamp + LLM metadata extraction into a single bound record. Prior art treats hashing, timestamping, and AI extraction as independent pipelines. Binding them to a single indexed record accessible via a public URL is the inventive combination.
  2. Domain-conditional extraction prompt + scoring rule selection from a shared library. Prior art uses either generic extraction (one-size-fits-all) or bespoke vertical products (cannabis-only, academic-only). A system that switches domain via a parameter is the inventive architecture.
  3. Hybrid deterministic-stochastic forensic scoring. Using stochastic LLMs to evaluate qualitative signals while still producing a deterministic, reproducible, explainable final verdict is novel in document authentication.
  4. Public verification page enabling third-party re-verification without operator trust. Prior art verification endpoints either require authentication or simply display text; a public UUID-addressed page that exposes the stored hash for independent re-computation is novel.
  5. Integrated signed attestation certificate suitable for evidentiary use under FRE 902(13)/902(14). The combination of the technical output with self-authentication-ready legal packaging is novel; prior art separates technical timestamping from legal certification.
  6. Asynchronous processing with real-time state broadcasting. Prior art authentication services are synchronous or hide async processing; exposing the state transitions via publish-subscribe for client-side UX is novel in this context.

Counsel should draft independent method claims and system claims that cover the combination of all six core elements, with dependent claims addressing: alternative hash algorithms; alternative timestamp sources; alternative LLM providers; alternative scoring axes and decision-tree structures; alternative content types; alternative blockchain anchoring schemes; alternative QR encoding; alternative broadcasting protocols.
